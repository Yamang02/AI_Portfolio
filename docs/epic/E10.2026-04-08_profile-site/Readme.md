# Epic E10: profile-site

## 목표

- `profile/` 디렉토리가 모노레포 내에 독립 Vite 프로젝트로 존재하고, 별도 도메인으로 서빙된다.
- `content.json`으로 관리되는 프로필 콘텐츠(identity + works)가 페이지에 렌더링된다.
- 프로필 페이지가 세 정체성(문제의식 / 구현가능성 / 영감) + 교차 결과물 + CTA 구조로 완성된다.

## 배경 / 맥락

### 현재 상태

- 포트폴리오 사이트(`frontend/`)는 모든 프로젝트·경력·학력을 나열하는 구조로, 클라이언트/협업 제안자를 위한 소개 목적에는 맞지 않는다.
- 나를 가장 잘 설명하는 경험을 큐레이션해서 소개하는 별도 채널이 없다.

### 문제

기존 포트폴리오는 전부를 보여주는 방향이라 클라이언트·협업 제안자에게 "이 사람이 누구인가"를 빠르게 전달하지 못한다.

## 특이점

- **디자인 방향**: 교사(문제의식) · 기술자(구현가능성) · 예술가(영감) 세 정체성을 병렬 배치하고, 이 세 정체성이 교차해 나온 결과물들을 하단에 배치한다. CTA는 AI 포트폴리오 사이트로 연결한다.
- **소스 분리**: 기존 `frontend/`와 완전히 독립된 `profile/` Vite 프로젝트로 모노레포 내에 추가한다. 같은 레포에 두어 향후 백엔드 연결 시 마이그레이션 비용을 낮춘다.
- **콘텐츠 관리**: 현재는 `profile/public/content.json` 정적 파일로 관리한다. 편집 빈도가 낮고 어드민 UI 구축 비용이 이득보다 크기 때문. 향후 편집 빈도가 높아지면 백엔드 API로 교체 가능하도록 데이터 fetch 레이어를 추상화한다.
- **백엔드 의존 없음 (현재)**: CORS 업데이트 불필요. 백엔드 API 호출 없음.
- **디자인 선행 필요**: P02·P03은 시각 디자인이 별도 에픽에서 확정된 이후 진행한다. P01은 디자인과 무관하게 선행 가능하다.
- **CTA는 하드코딩**: 포트폴리오 URL은 변경 빈도가 없으므로 프론트엔드 상수로 관리한다.
