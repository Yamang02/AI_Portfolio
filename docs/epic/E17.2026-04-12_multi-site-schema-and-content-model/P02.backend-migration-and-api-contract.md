# P02: backend-migration-and-api-contract

## 목표

- `project_technical_cards` 도입과 `projects.readme` 제거를 안전하게 마이그레이션한다.
- 프로젝트 상세 조회 계약에 `projectOverviewArticle`, `technicalCards`를 추가한다.
- `portfolio`/`business` 사이트 컨텍스트 enum 계약을 백엔드 응답에서 일관되게 유지한다.

## 구현 상세

### 전제 조건

- P01의 스키마 설계가 고정되어 있어야 한다.
- E16의 Option B 전략(`project_visibility.site` enum 유지)을 변경하지 않는다.
- 로컬 DB 기준선(2026-04-12) 확인:
  - `projects` 13건
  - `readme` 비어있지 않은 데이터 10건
  - `project-overview` 아티클 0건

### 접근 방식

1. DB 마이그레이션
- `project_technical_cards` 테이블/인덱스 생성
- 기존 `projects.readme` -> `articles(category='project-overview')` 데이터 이관
- 이관 검증(10건 전량) 후 `projects.readme` 제거
- `articles.business_id` 길이 제한(`VARCHAR(20)`) 때문에 overview business id는 `ovw-{project.business_id}` 규칙 사용

2. 백엔드 모델/매퍼 정렬
- `ProjectTechnicalCardJpaEntity` 추가
- 프로젝트 상세 응답 매퍼에서 `projectOverviewArticle` 단건 조회 + `technicalCards` 목록 조회 결합
- 정렬 규칙 고정: `is_pinned DESC, sort_order ASC`

3. API 계약 확정
- `GET /api/projects/{id}` 응답 스키마 확장
- Admin CRUD에서 Technical Card 생성/수정/삭제 지원
- 카드-아티클 연결 규칙 고정
  - `article_id`는 같은 프로젝트 문서만 허용
  - 참조 아티클 삭제 시 `SET NULL` 동작을 API 응답에서 일관 처리

### 변경 파일(예정)

- `backend` 마이그레이션 SQL
  - `backend/src/main/resources/db/migration/V006__add_project_technical_cards_and_migrate_project_overview.sql` (non-breaking)
- `database` 프로덕션 수동 SQL
  - `database/production/2026-04-12_E17_gcp_cutover.sql` (GCP 이전 시 1회 통합 실행용)
- 프로젝트/아티클/기술카드 관련 JPA 엔티티, 리포지토리, 매퍼, 서비스, 컨트롤러
