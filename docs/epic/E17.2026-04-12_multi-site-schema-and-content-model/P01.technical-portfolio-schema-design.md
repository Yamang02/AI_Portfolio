# P01: technical-portfolio-schema-design

> 본 문서는 E16의 `D04.technical-portfolio-schema-design.md`에서 승격된 후속 설계 문서다.

## 기술 포트폴리오를 위한 DB 스키마 재설계

**날짜**: 2026-04-12
**결정 유형**: DB 스키마 설계
**영향도**: 높음
**상태**: 설계 완료

---

## 배경

현재 포트폴리오 사이트를 **기술자의 포트폴리오**로 전환하면서, 단순히 "무엇을 만들었는가"를 넘어 **"어떻게 문제를 해결했는가"**를 보여주는 구조가 필요함.

### 기존 구조의 한계
1. **프로젝트 중심 구조**: 프로젝트 단위로만 정보 관리
2. **Article은 독립적**: 프로젝트와 연결되지만, 깊이 있는 기술 맥락 부족
3. **관리 복잡도**: displayOrder, portfolio_mode 등 관리 필드 과다

---

## 목표

**선택된 프로젝트**에 대해 다음 정보를 체계적으로 관리:
1. **문제 정의**: 어떤 기술적 문제가 있었는가
2. **분석**: 왜 그 문제가 발생했는가
3. **해결**: 어떻게 해결했는가
4. **상세 문서** (선택): Article로 심도있는 내용 제공

---

## 핵심 설계 원칙

1. **YAGNI (You Aren't Gonna Need It)**: 지금 필요한 것만 구현
2. **단순성**: 관리 필드 최소화, 콘텐츠로 상태 판단
3. **명확한 책임 분리**: 카드(요약) vs Article(상세)
4. **재사용성**: UI 컴포넌트 재사용 최대화

---

## 최종 스키마 구조

### 1. 프로젝트 기술 카드 (Project Technical Cards)

**새 테이블**: `project_technical_cards`

```sql
CREATE TABLE project_technical_cards (
    id BIGSERIAL PRIMARY KEY,
    business_id VARCHAR(50) UNIQUE NOT NULL,
    project_id BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,

    -- 카드 기본 정보
    title VARCHAR(255) NOT NULL,
    category VARCHAR(50) NOT NULL, -- architecture | troubleshooting | feature

    -- 문제-분석-해결 (카드에 표시)
    problem_statement TEXT NOT NULL,    -- 문제 정의 (2-3줄)
    analysis TEXT,                       -- 분석 (2-3줄, 선택)
    solution TEXT NOT NULL,              -- 해결 방법 (2-3줄)

    -- Article 연결 (선택)
    article_id BIGINT REFERENCES articles(id) ON DELETE SET NULL,

    -- 노출 순서 관리
    is_pinned BOOLEAN DEFAULT FALSE,  -- 중요 카드 고정
    sort_order INTEGER DEFAULT 0,     -- 같은 그룹 내 정렬

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_technical_cards_project ON project_technical_cards(project_id);
CREATE INDEX idx_technical_cards_pinned ON project_technical_cards(is_pinned);
CREATE INDEX idx_technical_cards_category ON project_technical_cards(category);
```

**특징**:
- 프로젝트에 **종속** (projectId 필수)
- Article과 **선택적 연결** (간단한 이슈는 카드만, 복잡한 이슈는 카드 + Article)
- PINNED + sort_order로 노출 순서 관리 (일반적인 패턴)

### 2. Article 역할 변경

**기존 `articles` 테이블 유지**, 용도 명확화:

**Article 사용 패턴**:
1. **프로젝트 개요** (project-overview) - 프로젝트 상세 페이지에 **인라인 렌더링**
2. **독립 문서** (tutorial, insight, sns-posts) - projectId 없음
3. **기술 카드의 상세 문서** - technical_cards를 통해 연결
4. **프로젝트 문서** (development-timeline) - projectId 있음, 카드 없음

**카테고리**:
```sql
-- 'project-overview'      : 프로젝트 소개 (Readme 대체, 인라인 렌더링)
-- 'tutorial'              : 튜토리얼 (독립)
-- 'insight'               : 기술적 인사이트 (독립)
-- 'sns-posts'             : SNS 포스트 (독립)
-- 'development-timeline'  : 개발 과정 (프로젝트 문서, 카드 없음)
```

**스키마 변경**: articles 테이블은 기존 그대로 유지 (변경 없음)

### 3. Projects 테이블 변경

```sql
-- readme 필드 제거 (후행 breaking 단계에서만 수행)
-- 선행 조건: readme -> project-overview 이관/검증 완료
ALTER TABLE projects
DROP COLUMN readme;
```

**이유**:
- Readme를 `project-overview` Article로 관리
- Private/Public repo 상관없이 동작
- Admin에서 통합 관리 (모든 content가 Article)

## 도메인 전략 연동 기준 (E16-P02 반영)

`project_visibility.site`는 실제 도메인 문자열이 아니라 **사이트 컨텍스트 enum**으로 관리한다.

- `portfolio`: 개인 포트폴리오 맥락 노출
- `business`: 야망솔루션 비즈니스 맥락 노출

과도기 매핑(현재):
- `portfolio` → `www.yamangsolution.com` (임시 운영)
- `business` → 미오픈

목표 매핑(컷오버 후):
- `portfolio` → `portfolio.yamang02.com` 또는 `blog.yamang02.com` (개인 사이트 성격에 따라 선택)
- `business` → `yamangsolution.com`

스테이징 매핑(컷오버 후):
- `portfolio` → `staging.portfolio.yamang02.com` 또는 `staging.blog.yamang02.com`
- `business` → `staging.yamangsolution.com` (또는 추후 확정된 비즈니스 스테이징 도메인)

예시:
```sql
project_visibility (
  project_id,
  site: 'portfolio' | 'business',
  is_visible: boolean,
  custom_title,
  custom_description
)
```

// project.types.ts

## 프론트엔드 UI 구조

### 최종 프로젝트 상세 페이지

```
프로젝트 상세 페이지
├── 개요 (Overview)
│   ├── 프로젝트 설명
│   ├── 스크린샷
│   └── 기술 스택
│
├── 프로젝트 소개 (project-overview Article을 인라인 렌더링)
│   └── Article의 content를 직접 표시 (Readme 대체)
│
└── 기술 노트 (Technical Notes) - 기술 카드가 있을 때만 표시
    ├── TOC 아코디언 (카드 목차)
    │   ├── [PINNED] 카드 목록
    │   └── [일반] 카드 목록
    │
    └── 카드 리스트
        ├── PINNED 카드들
        └── 일반 카드들
```

### UI 구현 (심플한 디자인)

```tsx
<ProjectDetailPage>
  {/* 개요 섹션 */}
  <section id="overview">
    <SectionTitle level="h2">개요</SectionTitle>
    <Description>{project.description}</Description>
    <Screenshots images={project.screenshots} />
    <TechStackList technologies={project.technologies} />
  </section>

  {/* project-overview Article 인라인 렌더링 */}
  {project.projectOverviewArticle && (
    <section id="project-overview">
      <SectionTitle level="h2">프로젝트 소개</SectionTitle>
      <MarkdownRenderer
        content={project.projectOverviewArticle.content}
      />
    </section>
  )}

  {/* 기술 노트 섹션 (카드가 있을 때만) */}
  {technicalCards && technicalCards.length > 0 && (
    <section id="technical-notes">
      <SectionTitle level="h2">기술 노트</SectionTitle>

      {/* TOC 아코디언 */}
      <Accordion defaultExpanded={false}>
        <AccordionSummary>
          목차 ({technicalCards.length}개 노트)
        </AccordionSummary>
        <AccordionDetails>
          {/* PINNED 그룹 */}
          {pinnedCards.length > 0 && (
            <TocGroup>
              <TocGroupTitle>주요 결정</TocGroupTitle>
              {pinnedCards.map(card => (
                <TocItem onClick={() => scrollTo(`card-${card.businessId}`)}>
                  <CategoryBadge mini>{card.category}</CategoryBadge>
                  {card.title}
                </TocItem>
              ))}
            </TocGroup>
          )}

          {/* 일반 그룹 */}
          {normalCards.length > 0 && (
            <TocGroup>
              <TocGroupTitle>기타 노트</TocGroupTitle>
              {normalCards.map(card => (
                <TocItem onClick={() => scrollTo(`card-${card.businessId}`)}>
                  <CategoryBadge mini>{card.category}</CategoryBadge>
                  {card.title}
                </TocItem>
              ))}
            </TocGroup>
          )}
        </AccordionDetails>
      </Accordion>

      {/* 카드 리스트 (아이콘 없음, 심플) */}
      <CardList>
        {pinnedCards.map(card => (
          <TechnicalCard
            key={card.businessId}
            id={`card-${card.businessId}`}
            card={card}
            isPinned={true}
          />
        ))}

        {normalCards.map(card => (
          <TechnicalCard
            key={card.businessId}
            id={`card-${card.businessId}`}
            card={card}
          />
        ))}
      </CardList>
    </section>
  )}
</ProjectDetailPage>
```

## 마이그레이션 계획

### 로컬 PostgreSQL 기준선 (2026-04-12)

`docker compose exec postgres psql -U dev_user -d ai_portfolio` 조회 결과:

- `projects` 총 13건
- `readme` 비어있지 않은 프로젝트 10건
- `readme` 비어있는 프로젝트 3건
- `articles.category = 'project-overview'` 기존 데이터 0건
- `ovw-*` 패턴 `business_id` 기존 데이터 0건
- `articles.business_id` 길이 제한: `VARCHAR(20)` (로컬 조회 기준)

결론: `readme` 제거는 즉시 DROP이 아니라, **10건 이관 완료 검증 이후 DROP**으로 진행해야 한다.

### Phase 1: 스키마 변경 (V006 마이그레이션, non-breaking)

```sql
-- 1. project_technical_cards 테이블 생성
CREATE TABLE project_technical_cards (
    id BIGSERIAL PRIMARY KEY,
    business_id VARCHAR(50) UNIQUE NOT NULL,
    project_id BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    title VARCHAR(255) NOT NULL,
    category VARCHAR(50) NOT NULL,
    problem_statement TEXT NOT NULL,
    analysis TEXT,
    solution TEXT NOT NULL,
    article_id BIGINT REFERENCES articles(id) ON DELETE SET NULL,
    is_pinned BOOLEAN DEFAULT FALSE,
    sort_order INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_technical_cards_project ON project_technical_cards(project_id);
CREATE INDEX idx_technical_cards_pinned ON project_technical_cards(is_pinned);
CREATE INDEX idx_technical_cards_category ON project_technical_cards(category);

-- 2. articles 테이블은 기존 유지 (변경 없음)
-- 3. projects.readme는 아직 유지 (데이터 이관 전 DROP 금지)
```

### Phase 2: 데이터 마이그레이션

1. 기존 `readme` 데이터를 `project-overview` Article로 변환한다.
2. 각 프로젝트의 readme -> Article 생성 후 건수를 검증한다.
3. 기존 article 데이터는 그대로 유지한다.
4. 새로운 기술 카드는 Admin에서 수동 생성한다.

**마이그레이션 스크립트 예시**:
```sql
-- idempotent insert: 기존 project-overview가 없는 프로젝트만 생성
INSERT INTO articles (business_id, title, content, category, project_id, status, created_at, updated_at)
SELECT
    CONCAT('ovw-', p.business_id),
    CONCAT(p.title, ' 소개'),
    p.readme,
    'project-overview',
    p.id,
    'published',
    NOW(),
    NOW()
FROM projects p
WHERE p.readme IS NOT NULL
  AND btrim(p.readme) <> ''
  AND NOT EXISTS (
      SELECT 1
      FROM articles a
      WHERE a.project_id = p.id
        AND a.category = 'project-overview'
  )
  AND NOT EXISTS (
      SELECT 1
      FROM articles a2
      WHERE a2.business_id = CONCAT('ovw-', p.business_id)
  );
```

**검증 쿼리 (DROP 전 필수)**:
```sql
-- 1) readme 대상 건수와 생성된 overview article 건수가 같은지 확인
SELECT
    COUNT(*) FILTER (WHERE p.readme IS NOT NULL AND btrim(p.readme) <> '') AS source_readme_count,
    COUNT(*) FILTER (WHERE a.id IS NOT NULL) AS migrated_article_count
FROM projects p
LEFT JOIN articles a
  ON a.project_id = p.id
 AND a.category = 'project-overview';

-- 2) 미이관 프로젝트 확인 (0건이어야 함)
SELECT p.id, p.business_id, p.title
FROM projects p
WHERE p.readme IS NOT NULL
  AND btrim(p.readme) <> ''
  AND NOT EXISTS (
      SELECT 1
      FROM articles a
      WHERE a.project_id = p.id
        AND a.category = 'project-overview'
  );
```

### Phase 3: `projects.readme` 제거 (breaking)

```sql
ALTER TABLE projects
DROP COLUMN readme;
```

검증 기준:

- `source_readme_count = migrated_article_count`
- 미이관 프로젝트 0건

위 기준이 충족되지 않으면 `DROP COLUMN readme`를 실행하지 않는다.

### Phase 4: Backend 구현

1. **Entity 생성**
   - `ProjectTechnicalCardJpaEntity` 생성
   - `ProjectJpaEntity`: readme 필드 제거

2. **Mapper 업데이트**
   - 프로젝트 응답에 `projectOverviewArticle` 포함
     - `category='project-overview' AND project_id={id}` 조회
   - 프로젝트 응답에 `technicalCards` 포함
     - 정렬 로직: `is_pinned DESC, sort_order ASC`

3. **API 확장**
   - `GET /api/projects/{id}`: projectOverviewArticle, technicalCards 포함
   - Admin API: Technical Card CRUD
   - Admin API: project-overview Article CRUD

### Phase 5: Frontend 구현

1. **타입 정의**
   - `ProjectTechnicalCard` 타입 생성
   - `Project` 타입: readme 제거, projectOverviewArticle, technicalCards 추가
   - `Article` 타입: category에 'project-overview' 추가

2. **컴포넌트 구현**
   - `<TechnicalCard />` - 카드 컴포넌트 (아이콘 없음)
   - `<TechnicalNotesTOC />` - TOC 아코디언
   - `<ProjectDetailPage />` 업데이트
     - project-overview Article 인라인 렌더링 섹션 추가
     - 섹션 타이틀에서 아이콘 모두 제거

3. **Admin 페이지**
   - Technical Card CRUD 화면
   - PINNED, sortOrder 관리 UI
   - project-overview Article 편집 UI

| **카드 vs Article** | 카드(요약) + Article(상세, 선택) 분리 | 명확한 책임 분리, 유연성 |
| **Readme** | project-overview Article로 관리, 인라인 렌더링 | Private repo 지원, 통합 관리 |
| **버전/마일스톤** | 제거, 카드로 대체 | YAGNI 원칙, 과도한 설계 방지 |
| **카드 노출 순서** | is_pinned + sort_order | 일반적 패턴, 관리 용이 |
| **UI 구조** | Overview + 프로젝트 소개 + 기술 노트 | 핵심에 집중, 명확한 흐름 |
| **TOC** | 아코디언, PINNED/일반 구분 | 전체 파악 용이, 빠른 이동 |
| **아이콘** | 제거 (텍스트만) | 심플한 디자인 |

### 🎯 장점

1. **단순성**: 최소한의 테이블과 필드로 목표 달성
2. **명확성**: 카드(요약) vs Article(상세) 역할 분리
3. **유연성**: 간단한 이슈는 카드만, 복잡한 이슈는 Article 추가
4. **확장성**: 새 카테고리 추가 용이
5. **관리 용이**: PINNED로 중요도 관리, 자동 정렬
6. **통합 관리**: 모든 content가 Article (Admin에서 통합 관리)
7. **Private repo 지원**: Readme도 DB에 저장되어 Public/Private 무관

### ⚠️ 주의사항

1. **Breaking Change**: readme 필드 제거는 이관/검증 완료 후 마지막 단계에서만 수행
2. **새 Entity**: ProjectTechnicalCard 구현 필요
3. **Admin UI**: 카드 관리 화면, project-overview Article 편집 UI 구현 필요
4. **렌더링 구분**: project-overview는 인라인, Technical Cards는 카드 형태
