# P02: admin-project-technical-card-contract

## 목표

- E17 스키마 기준으로 Admin 프로젝트 편집의 입출력 계약을 확정한다.
- `readme` 중심 계약을 제거하고 `projectOverviewArticle` + `technicalCards` 구조를 표준화한다.

## 구현 상세

### 전제 조건

- P01 FSD 정리가 완료되어야 한다.
- E17 DB 마이그레이션(`project_technical_cards` 테이블)이 반영되어 있어야 한다.
- 기존 `/api/admin/projects` 사용 흐름을 깨지 않는 non-breaking 전환 전략이 필요하다.

### 접근 방식

1. Admin Project 응답/요청 DTO 확장
- `technicalCards` 필드(카드 본문, 정렬, 핀, 연결 article reference)를 명시
- `projectOverviewArticle` 연결 규칙 명시

2. 도메인/유스케이스 경계 확정
- 카드 수정 책임이 `ManageProjectUseCase` 내부인지 별도 유스케이스인지 결정
- 트랜잭션 경계와 정렬 규칙(`isPinned DESC, sortOrder ASC`) 고정

3. 호환성 전략
- 전환 기간 동안 `readme` 사용 경로를 deprecated로 표시
- main/admin의 새 계약 적용 완료 후 제거 플랜 확정

## 구현 메모 (2026-04-13)

- 저장 경계:
  - `Project` Aggregate가 `technicalCards`를 소유하고 기존 `PortfolioRepository` + JPA 매핑에서 함께 저장한다.
  - 별도 `TechnicalCardRepository` 포트는 만들지 않고, Aggregate 경계 일관성을 우선한다.
- API:
  - `GET /api/admin/projects/{id}/technical-cards`
  - `PUT /api/admin/projects/{id}/technical-cards`
- `projectOverviewArticle` 규칙:
  - category: `project-overview`
  - businessId: `ovw-{projectBusinessId}` (E17 컷오버 SQL 규칙)
- `readme`는 전환 호환용으로 유지하되 deprecated 마킹 적용.
