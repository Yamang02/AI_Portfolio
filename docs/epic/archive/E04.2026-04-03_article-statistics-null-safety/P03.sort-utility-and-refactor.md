# P03: sort 유틸리티 도입 및 전체 패턴 리팩터

## 목표

문자열 필드 sort 비교 로직을 `shared/utils/sortUtils.ts`에 집중하고, 코드베이스 내 모든 unsafe sort 패턴이 해당 유틸을 사용하도록 교체한다.  
P01에서 임시 수정된 두 파일도 유틸 기반으로 재작성되어 일관된 패턴을 가진다.

## 구조적 원인 분석

### 반복 발생 이유

| 요인 | 설명 |
|------|------|
| 타입과 런타임의 불일치 | 응답 타입에서 `string`(required)으로 선언하지만 DB/API 실제 값은 null 가능. TypeScript가 컴파일 타임에 잡지 못함. |
| 공유 유틸 부재 | sort comparator를 호출 지점마다 인라인으로 작성. 방어 처리 여부가 작성자에 따라 달라짐. |
| 패턴 파편화 | `a.name.localeCompare(b.name)`, `(a ?? '').localeCompare(b ?? '')`, `|| 0`, `!` 단언 등 같은 문제를 제각각 처리. |

### 현재 unsafe 지점 전수 조사 결과

| 파일 | 라인 | 패턴 | 위험도 |
|------|------|------|--------|
| `main/pages/ArticleListPage.tsx` | 118 | `a.title.localeCompare(b.title)` | **크래시 재현 중** |
| `main/entities/tech-stack/ui/TechStackList.tsx` | 110, 117 | `a.name.toLowerCase()`, `a.name.localeCompare(b.name)` | 잠재적 크래시 |
| `main/pages/ProfilePage/components/CareerTimeline.tsx` | 60–64 | `find()!` 비-null 단언 — 항목 미발견 시 undefined.startDate | 잠재적 크래시 |

나머지 sort 지점(날짜 계산, 숫자, DOM IntersectionObserver)은 이미 안전하게 처리됨.

## 구현 상세

### 1. `sortUtils.ts` 신규 생성

**위치**: `frontend/src/shared/utils/sortUtils.ts`

제공할 함수:

```typescript
/** null/undefined를 빈 문자열로 처리하여 문자열 두 값을 비교 */
export const compareStrings = (
  a: string | null | undefined,
  b: string | null | undefined,
): number => (a ?? '').localeCompare(b ?? '');
```

단일 책임: 문자열 null-safe 비교. 날짜·숫자 비교는 이미 각 호출 지점에서 안전하게 처리 중이므로 이 유틸 범위에 포함하지 않는다.

### 2. 호출 지점 교체

**`ArticleListPage.tsx`**

```typescript
// before (P01 임시 수정)
.sort((a, b) => (a.title ?? '').localeCompare(b.title ?? ''))

// after
.sort((a, b) => compareStrings(a.title, b.title))
```

**`TechStackList.tsx (entities)`**

```typescript
// before (P01 임시 수정)
.sort(({ name: nameA = '' }, { name: nameB = '' }) => {
  const aIsCore = coreTechs.has(nameA.toLowerCase());
  const bIsCore = coreTechs.has(nameB.toLowerCase());
  if (aIsCore && !bIsCore) return -1;
  if (!aIsCore && bIsCore) return 1;
  return nameA.localeCompare(nameB);
})

// after
.sort(({ name: nameA }, { name: nameB }) => {
  const aIsCore = coreTechs.has((nameA ?? '').toLowerCase());
  const bIsCore = coreTechs.has((nameB ?? '').toLowerCase());
  if (aIsCore && !bIsCore) return -1;
  if (!aIsCore && bIsCore) return 1;
  return compareStrings(nameA, nameB);
})
```

**`CareerTimeline.tsx`** — 정렬 키를 항목 생성 시 계산하고 `shared/utils/flexibleMonthDate.ts`로 파싱 위임

- 항목 빌드 시 `sortKeyMs: startOfMonthTimeMs(startDate)` 저장 → 정렬 comparator에서 `find()` 불필요, CC-01-05에 맞는 `left`/`right` 비교만 유지.

### 3. API 응답 타입 수정 — nullable 필드 명시

**왜 API 클라이언트가 아닌가:**  
`apiClient.callApi<T>`는 HTTP 전송(재시도·에러·baseURL) 책임만 가진다. 데이터 모양 검증을 여기서 하려면 모든 응답 스키마를 알아야 해 결합도가 높아진다.  
올바른 위치는 **타입 정의**다. `string`을 `string | null`로 바꾸면 TypeScript가 컴파일 타임에 모든 소비 지점을 강제한다.

```typescript
// frontend/src/main/entities/article/api/articleApi.ts (또는 통계 응답 타입 정의 위치)
series: Array<{
  seriesId: string;
  seriesTitle: string | null;  // DB에서 null 가능
  count: number;
}>
```

이 수정 하나로 `ArticleListPage`의 sort 코드가 `compareStrings` 없이는 컴파일되지 않게 된다 — 미래 회귀 방지.
