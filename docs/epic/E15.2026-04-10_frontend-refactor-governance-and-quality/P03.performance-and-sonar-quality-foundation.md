# P03: performance-and-sonar-quality-foundation

## 목표

- SonarQube 점수를 올리기 위한 구조적 품질 작업과 코드 정리를 병행한다.
- 반복 규칙 대량 구간과 hotspot 파일을 정리해 유지보수성과 성능을 함께 개선한다.
- 테스트 baseline을 확장해 refactor 안전성을 확보한다.
- 측정 가능한 before/after evidence를 남긴다.

## 현재 baseline

- `docs/issues/sonarqube/2026-03-27-sonarqube-open-issues-summary.md` 기준 frontend open issues: 336
- CRITICAL 1건: `src/main/features/easter-eggs/effects/matrix/MatrixEffect.tsx`
- 반복 규칙 상위: `typescript:S7764`, `typescript:S7773`, `typescript:S1128`, `typescript:S7735`, `typescript:S2933`
- 테스트 파일 수: 2
- `frontend/sonar-project.properties`는 Storybook 파일을 production scope에서 분리하지 않고 있다.

## 구현 상세

### 1. Sonar scope hygiene

- production quality gate와 story/demo artifact를 분리한다.
- `frontend/sonar-project.properties`에서 `*.stories.*` 등 non-production 파일의 분석 범위를 조정한다.
- exclusions는 점수 가리기가 아니라 production code 평가 정확도 향상 목적에 한정한다.

### 2. 반복 이슈 정리 wave

- low-risk 반복 규칙은 일괄 정리한다.
- 후보: `globalThis`, `Number.isNaN`, unused imports, readonly, negated condition 정리
- rule count가 높은 순으로 처리하되, behavior change 위험이 낮은 군부터 정리한다.

### 3. Hotspot refactor

우선 후보:

- `frontend/src/main/pages/ArticleListPage.tsx`
- `frontend/src/main/pages/ProjectDetailPage/ProjectDetailPage.tsx`
- `frontend/src/admin/pages/ProjectEdit.tsx`
- `frontend/src/shared/ui/markdown/MarkdownRenderer.tsx`
- `frontend/src/shared/lib/analytics.ts`
- `frontend/src/main/features/easter-eggs/effects/matrix/MatrixEffect.tsx`

정리 방향:

- page orchestration, data fetching, rendering concern을 분리한다.
- complexity가 높은 함수는 helper/hook/section component로 분해한다.
- refactor 후 테스트를 추가해 regression을 막는다.

### 4. Performance 정리

- `profile`의 CDN/inline runtime 의존을 제거하고 build artifact 기반으로 정리한다.
- route-level chunk와 asset loading을 다시 점검한다.
- 불필요한 inline script, 중복 렌더링 경로, 대형 page component를 줄여 초기 로딩과 유지보수 비용을 함께 낮춘다.

### 5. Evidence

- `npm run test:coverage`
- `npm run build`
- Sonar 재분석 결과
- 필요 시 bundle or page-size 비교 로그

## 체크리스트

- [ ] Storybook/demo 파일이 production Sonar scope에서 분리되었다.
- [ ] frontend Sonar CRITICAL issue가 0건이 되었다.
- [ ] 상위 반복 규칙 구간이 의미 있게 축소되었다.
- [ ] hotspot 페이지 또는 유틸이 더 작은 단위로 분해되었다.
- [ ] 테스트 파일 수와 coverage baseline이 증가했다.
- [ ] `profile`과 `frontend`의 빌드/분석 evidence가 남았다.
- [ ] 점수 개선이 exclusion 편법이 아니라 구조 개선 결과임을 설명할 수 있다.
