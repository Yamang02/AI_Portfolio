# P01: runtime-boundary-and-profile-consolidation

## 목표

- `frontend`와 `profile`의 canonical responsibility를 명확히 정의한다.
- `profile/index.html`이 canonical surface임을 확정하고, 연결되지 않은 `profile/src/` dead code를 정리한다.
- `main`과 `admin`을 추후 분리 가능한 독립 앱으로 취급하는 FSD 6레이어 구조를 확정하고, alias/lint/doc 규칙을 이 구조에 맞춘다.
- `frontend` 내부 `/profile` 의존을 제거하고, `yamangsolution.com` 런타임의 기본 진입 정책을 문서화한다.
- top-level `shared/`를 audit해 main 전용 / admin 전용 / 공용으로 분리한다.

## 구현 상세

### 1. Runtime ownership 확정

- `frontend/src/main`은 `portfolio-public` 독립 앱이다.
- `frontend/src/admin`은 `portfolio-admin` 독립 앱이다.
- `profile`은 개인 profile site 단일 surface를 담당한다.
- `main`과 `admin`은 추후 별도 repo로 분리될 수 있도록, 상호 직접 import를 금지한다.

### 2. `frontend` 목표 구조

```text
frontend/src/
  main/
    app/
    pages/
    widgets/
    features/
    entities/
    shared/          ← main 전용 유틸
  admin/
    app/
    pages/
    widgets/
    features/
    entities/
    shared/          ← admin 전용 유틸
  design-system/     ← cross-app UI 전용
  shared/            ← 진짜 공용만 (최소화)
```

- 각 앱(`main`, `admin`) 내부에서 FSD 6레이어 downward dependency를 적용한다.
- cross-app import는 `design-system/` 또는 top-level `shared/`를 통해서만 허용한다.
- top-level `shared/`는 양쪽 앱이 실제로 사용하는 것만 남긴다.

### 3. `profile` 목표 구조

- `profile/index.html`이 canonical rendering surface다. static HTML + Tailwind CDN 방식을 유지한다.
- `profile/src/emailjs-config.ts`는 제거하고, Vite HTML env 치환(`%VITE_*%`)으로 대체한다. index.html 내 인라인 script에서 직접 env 값을 참조한다.
- `profile/src/App.tsx`, `profile/src/api/`, `profile/src/main.tsx`, `profile/src/types/`는 index.html에 연결되지 않은 dead code다. 제거한다.

### 4. 정리 대상

| 대상 | 현재 문제 | P01 결과 |
|------|-----------|-----------|
| `frontend/src/main/app/MainAppRoutes.tsx` | `/profile` 유지, `/` redirect도 profile 기준 | `/profile` 라우트 유지 (SEO), root policy 문서화 |
| `frontend/src/main/pages/ProfilePage/` | 디렉터리명이 역할을 반영 못함 | `FounderPage/`로 rename (라우트 URL은 유지) |
| `profile/index.html` | inline script와 정적 UI가 canonical처럼 동작 | canonical임을 확정. dead code 제거 |
| `frontend/vite.config.ts` | 존재하지 않는 root alias 6개 보유 | 실제 구조 기준으로 정리 |
| `frontend/tsconfig.json` | vite.config와 paths 불일치 | 목표 구조와 일치하게 정리 |
| `frontend/.eslintrc.cjs` | 구 flat FSD alias 기준 규칙 | 독립 앱 구조 기반 import 규칙으로 갱신 |
| `docs/technical/architecture/fsd-refactoring.md` | 코드와 문서 사이 구조 드리프트 | E15 확정 구조로 재작성 |
| `frontend/src/shared/` | main 전용 코드가 대부분을 차지 | main/shared, admin/shared, 공용으로 분리 |

### 5. `shared/` 분리 기준

audit 결과 기준:

| 분류 | 해당 항목 |
|------|----------|
| **top-level `shared/` 유지** (양쪽 공용) | `types/api`, `article`, `config/queryCacheConfig`, `ui/seo/SeoHead`, `hooks/useDebounce` |
| **`main/shared/`로 이동** | `ui/page-transition`, `ui/chat`, `ui/modal`, `ui/LoadingScreen`, `ui/markdown`, `ui/page-meta`, `ui/tech-stack`, `api/apiClient`, `utils/sortUtils`, `utils/safeStringUtils`, `utils/flexibleMonthDate`, `lib/schema`, `lib/markdown`, `lib/useThemeInit`, `config/seo.config`, 나머지 hooks |
| **`admin/shared/`로 이동** | `api/upload-api` |

### 6. 작업 순서

1. `profile` dead code 제거 + emailjs → `%VITE_*%` 교체
2. `frontend` `/profile` 라우트 및 `ProfilePage/` 제거
3. `vite.config.ts`, `tsconfig.json`, `.eslintrc.cjs` 정리
4. `fsd-refactoring.md` + 아키텍처 문서 업데이트
5. `shared/` audit → `main/shared/`, `admin/shared/`, top-level 분리

## 결정 원칙

- `profile`과 `frontend` 사이 직접 import를 새로 만들지 않는다.
- `main`과 `admin` 사이 직접 import를 새로 만들지 않는다.
- `/profile`를 유지하는 타협안은 허용하지 않는다.
- `shared/` 분리는 TypeScript 컴파일 통과를 완료 기준으로 삼는다.
