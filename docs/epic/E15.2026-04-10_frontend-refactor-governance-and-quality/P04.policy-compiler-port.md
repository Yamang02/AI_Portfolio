# P04: policy-compiler-port

## 목표

- E15에서 검증된 결정 중 재사용 가능한 것만 `policy_compiler`에 이식한다.
- 제품 특화 결정과 basis/convention 수준 결정을 분리한다.
- 이후 다른 프로젝트에도 적용 가능한 frontend architecture / design token / quality governance 문서를 남긴다.

- `policy_compiler/docs/basis/convention/design-tokens/DT-01-design-token-foundations.md`

## 이식 대상

### 1. Frontend structure selection rationale

- 어떤 조건에서 full FSD 대신 runtime-first slim FSD를 선택하는가
- runtime boundary와 shared/design-system boundary를 어떻게 나누는가
- `main`/`admin`처럼 deploy/runtime가 다른 영역 사이 import를 어떻게 금지하는가

### 2. Token governance

- Primitive + Semantic + UI shell 우선 전략
- `--ds-*`, `--ui-*` naming과 allowed consumer 범위
- migration bridge/deprecation rule
- UI code direct hex/rgb 금지와 예외 범위

### 3. Quality governance

- production code와 story/demo/test artifact를 정적 분석에서 어떻게 분리할 것인가
- 구조 리팩토링과 Sonar score 개선을 함께 다룰 때 필요한 evidence 기준

## 이식하지 않을 항목

- `AI_PortFolio` 고유 runtime 이름(`main`, `admin`, `profile`)
- `yamang02.com`, `yamangsolution.com` 같은 도메인 정책
- 특정 페이지(`ArticleListPage`, `ProjectEdit`)나 특정 파일명
- 제품별 root route 정책과 profile UX 결정

## 산출물 기대치

- `AR-02-frontend-architecture.md`의 selection rationale 또는 예시 보강
- `DT-01`, `DT-02`, `DT-04`에 AI_PortFolio에서 검증된 naming/governance 보강
- 필요하면 `policy_compiler` 내부 별도 epic 또는 phase 문서 추가

## 체크리스트

- [ ] E15의 결정 중 product-specific rule과 reusable rule이 분리되었다.
- [ ] `policy_compiler`로 가져갈 architecture decision이 문서화되었다.
- [ ] `policy_compiler`로 가져갈 token governance decision이 문서화되었다.
- [ ] quality governance 이식 범위가 과도하지 않게 정리되었다.
- [ ] `policy_compiler` 문서 변경 후에도 AI_PortFolio 전용 결정은 이 저장소에 남아 있다.
