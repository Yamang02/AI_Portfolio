## 야망솔루션 사업 구조

### 두 가지 사업

1. **개발 외주** — 웹서비스/제품 개발 수주
2. **바이브코딩 강의** — AI를 활용한 코딩 강의 (현재 원격 화상강의 운영 중)

### 주요 타겟

- **강의**: 비개발자 (창업자, 소상공인, 1인 사업자 등) — 주력
- **강의 확장**: 개발자 대상 AI 활용 강의 — 2차
- **외주**: 웹페이지/서비스가 필요한 개인·사업자

### 핵심 메시지

> "내 사업 웹페이지, AI로 직접 만들 수 있습니다"

- 강의: 직접 만들고 싶은 사람
- 외주: 맡기고 싶은 사람
- 같은 고객이 두 서비스를 모두 소비 가능 (강의 → 외주 전환 or 병행)

### 현재 강의 상태

- 수강생 있음, 초기 단계 (피드백/후기 미확보)
- 수강 결과물: 웹페이지 구성
- 모객 방식: 커뮤니티 직접 활동 (비개발자 커뮤니티 중심)
- 커뮤니티 활동이 곧 신뢰 채널 → 사이트는 클로징 도구로 기능

### 교육 사업 방향

- 대면강의(시간 투입형) 지양, 교육자료 중심으로 전환 목표
- 교육자료는 **무료 제공** 방향으로 고민 중 → 신뢰/인지도 구축 + 모객 도구로 활용
- 대면강의는 프리미엄 옵션으로 선택적 운영
- 교육자료 형태 및 배포 방식 미결 (PDF/영상/패키지 등)

**사업 수익 구조 (목표)**
```
무료 교육자료   → 신뢰 형성, 커뮤니티 모객 도구
      ↓
프리미엄 대면강의  → 유료 (자료만으로 부족할 때)
개발 외주          → 유료 (직접 만들기 싫을 때)
```

## DB 설계에 미치는 영향

### 프로젝트 노출 이슈

같은 프로젝트 데이터가 두 사이트에서 다른 맥락으로 노출될 수 있음:
- `portfolio.yamang02.com` 또는 `blog.yamang02.com`: "소속 개발자로서 내가 한 것"
- `yamangsolution.com`: "이런 것도 만들 수 있습니다" 레퍼런스

### 필요한 스키마 변경

1. **`project_visibility` 테이블 추가** — 사이트별 노출 여부 및 가공 정보 관리
   ```sql
   project_visibility (
     project_id,
     site: 'portfolio' | 'business',
     is_visible: boolean,
     custom_title,
     custom_description
   )
   ```

2. **`projects.source` 재정의** — 현재 'github'/'local' (입력 출처) 의미로 사용 중
   - 비즈니스 맥락 축 추가 필요: `context` 컬럼 (`EMPLOYMENT` | `BUSINESS` | `PERSONAL`)

3. **yamangsolution.com 전용 신규 테이블** (미래)
   - `services` — 제공 서비스 목록
   - `inquiries` — 문의 내역
   - `lectures` — 강의 커리큘럼

### 보류 이유

DB/시스템 전체 구조 재설계가 선행되어야 함. 프론트엔드 분리도 DB 재설계 완료 후 진행.
