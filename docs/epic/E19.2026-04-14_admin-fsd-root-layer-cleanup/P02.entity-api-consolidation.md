# P02: entity-api-consolidation

## 목표

- `admin/api/adminAuthApi.ts`를 `admin/entities/auth/` entity로 이동한다.
- `admin/api/adminProjectApi.ts`와 `admin/entities/project/api/projectApi.ts`의 중복을 확인하고 하나로 통합한다.
- `admin/hooks/useProjects.ts`와 `admin/entities/project/api/useProjectQuery.ts`의 중복을 정리한다.
- 이동·통합 후 `admin/api/` 폴더가 완전히 삭제된다.

## 구현 상세

### auth entity 신설

`admin/entities/auth/` 구조를 생성한다:

```
admin/entities/auth/
├── api/
│   └── adminAuthApi.ts   (기존 admin/api/adminAuthApi.ts 이동)
├── model/
│   └── auth.types.ts     (기존 admin/features/auth/model/auth.types.ts에서 공통 타입 분리)
└── index.ts
```

- `admin/features/auth/model/auth.types.ts`는 feature-scoped 타입이므로 entity 이동 시 중복 주의.

### project API 통합

1. `adminProjectApi.ts`(레거시)와 `projectApi.ts`(E18 신설)의 인터페이스를 비교한다.
2. 필터 조회(`ProjectFilter`)와 기술카드 갱신(`updateProjectTechnicalCards`) 등 현재 페이지가 사용하는 계약을 기준으로 단일 API를 정의한다.
3. 단일 API 계약이 확정되기 전까지 기존 페이지(`ProjectList`, `ProjectEdit`)의 필터/상세/기술카드 동작을 깨지 않도록 한다.
4. 모든 참조가 단일 API로 이행된 뒤 남은 파일은 제거한다(임시 shim 허용, 장기 공존 금지).

### useProjects 중복 정리

1. `useProjects.ts`(레거시)와 `useProjectQuery.ts`(E18 신설)의 hook 목록을 비교한다.
2. `useProjectQuery.ts`에 누락된 기능(필터 기반 목록, 상세 조회 옵션, 기존 invalidation 의도)을 먼저 보완한다.
3. 기존 쿼리 키(`['admin-projects', filter]`, `['admin-project', id]`)와 신규 키(`PROJECT_KEYS`) 중 하나로 표준을 정하고 일괄 통일한다.
4. 페이지 단위 동작 확인 후 `useProjects.ts` 참조를 전부 교체하고 파일을 삭제한다.

### 변경 범위

- `adminAuthApi.ts`를 import하는 `admin/hooks/useAuth.ts` 경로 업데이트
- `adminProjectApi.ts`를 import하는 `admin/hooks/useProjects.ts`, pages, features 경로 업데이트
- `admin/api/` 폴더 최종 삭제
