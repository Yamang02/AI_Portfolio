## 핵심 원인 패턴

### 1) 컴포넌트/함수 과밀화로 인한 복잡도 증가 (`S3776`)
- **관찰**: 렌더 조건, 상태 전환, 애니메이션 로직, 데이터 변환이 단일 함수에 집중됨.
- **원인**: 기능 추가 시 주변 로직에 분기만 누적되고, 중간 추출 단계(도우미 함수/하위 컴포넌트)가 생략됨.
- **예방 기준**:
  - 렌더 블록이 2개 이상 조건 묶음을 가지면 하위 컴포넌트로 추출.
  - 동일 관심사(TOC 빌드, 배경 레이어 렌더, 타임라인 계산)는 순수 함수로 분리.

### 2) 접근성 이벤트 규칙 미준수 (`S1082`, `S6853`)
- **관찰**: 클릭 가능한 `div`/오버레이, 연결되지 않은 `label`이 다수 발견됨.
- **원인**: 마우스 중심 인터랙션 구현 후 키보드/스크린리더 계약이 누락됨.
- **예방 기준**:
  - 클릭 가능 요소는 기본적으로 `button` 사용.
  - `label`은 반드시 `htmlFor` + 대응 `id`를 갖게 설계.
  - 오버레이 닫기 패턴은 `onClick` + `onKeyDown(Enter/Space)`를 템플릿화.

### 3) 배열/표현식 단축 작성의 부작용 (`S4043`, `S6582`, `S3358`)
- **관찰**: 인라인 `sort`, 중첩 ternary, 불필요한 null 체크가 누적됨.
- **원인**: "짧은 코드" 우선 습관으로 의도 표현보다 압축 표기를 선호.
- **예방 기준**:
  - `sort`는 원본 변경이 의도되지 않으면 복사본에서 수행.
  - 조건이 2개 초과면 ternary 대신 명시적 변수/함수로 분리.
  - `a && a.b`는 `a?.b`로 통일.

### 4) 클래스 멤버 가변성 과대 선언 (`S2933`)
- **관찰**: 재할당되지 않는 멤버가 `readonly` 없이 선언됨.
- **원인**: 초기 작성 시 확장 가능성을 과대 가정.
- **예방 기준**:
  - 생성 후 변경 없는 멤버는 기본 `readonly`.
  - 클래스 리뷰 시 "재할당 여부"를 체크리스트에 포함.

## 이번 배치에서 반영한 변경 유형

- 복잡도 완화: 함수 분리, 하위 컴포넌트 추출
- 접근성 보강: 시맨틱 요소 전환, 키보드 이벤트 추가, 라벨 연결
- 표현식 정리: 중첩 조건식 분해, optional chaining 적용
- 안정성 강화: 불변 멤버 `readonly` 적용, 배열 정렬 시 원본 보존

## 재발 방지 체크리스트 (초안)

1. 클릭 가능한 요소가 `button`/`a`가 아닌가?
2. 모든 `label`이 대응 `id`와 연결되어 있는가?
3. 렌더/이펙트 함수에 분기와 루프가 과도하게 누적되지 않았는가?
4. 중첩 ternary를 변수로 분리할 수 없는가?
5. 원본 배열 변이(`sort`, `reverse`)가 의도된 것인가?
6. 클래스 멤버 중 `readonly` 가능한 항목이 남아 있지 않은가?
