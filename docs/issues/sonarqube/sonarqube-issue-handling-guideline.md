# SonarQube 이슈 처리 지침

> `docs/basis` 스타일(`Meta`, `Rules`, `Conflict Resolution`, `Design Decisions`)을 적용한 프로젝트용 이슈 처리 기준.
> 목적은 이슈 우선순위, 수정 방식, 검증 방식의 일관성을 유지하는 것이다.

---

## Meta

### 적용 범위

- `AI_Portfolio_frontend_local`, `AI_Portfolio_backend_local`의 SonarQube 오픈 이슈
- `docs/issues/sonarqube/*open-issues*.json` 스냅샷 기반 정기 처리 작업

### 전제

- 서버: `http://localhost:9000`
- 이슈 상태: `resolved=false` (OPEN)
- 우선 처리 범위: `BUG + (BLOCKER/CRITICAL/MAJOR)`

### 규칙 티어

| 티어 | 규칙 | 기준 | 최소 적용 |
|------|------|------|----------|
| Core | SQ-01-01 ~ SQ-01-05 | 우선순위/검증 일관성이 깨짐 | S |
| Guide | SQ-01-06 ~ SQ-01-07 | 문서 품질/추적성 영향 | M |

### 우선순위 체계

규칙 충돌 시 아래 순서를 따른다.

1. 안정성 영향도 (`BUG` > `BLOCKER/CRITICAL/MAJOR`)
2. 재현 가능성 (재현 가능 이슈 우선)
3. 변경 범위 최소화 (작은 수정 우선)

## Rules

### SQ-01-01. 처리 대상 필터를 고정한다

**Rule:** 한 사이클의 처리 대상은 `BUG + (BLOCKER/CRITICAL/MAJOR)`로 고정한다.

**Why:** 대상 필터가 매번 바뀌면 처리량/성과를 비교할 수 없고, 안정성 개선 추세가 왜곡된다.

**DO:**

- Sonar API 또는 스냅샷에서 `type=BUG` 또는 `severity in (BLOCKER, CRITICAL, MAJOR)`로 선별한다.
- 선별 결과를 처리 배치 문서에 고정한다.

**Exception:** 운영 장애 대응 등 긴급 상황에서 단일 이슈를 즉시 처리해야 할 경우, 범위 외 이슈를 먼저 처리할 수 있다.

---

### SQ-01-02. 이슈 단위를 최소 변경으로 처리한다

**Rule:** 이슈는 “1 이슈 = 1 최소 수정” 원칙으로 처리한다.

**Why:** 여러 변경을 한 번에 섞으면 효과 검증이 불가능해지고 회귀 원인 추적이 어려워진다.

**DO:**

- 이슈 키(`issue.key`) 기준으로 작업 단위를 쪼갠다.
- 리팩터링/구조개선은 별도 배치로 분리한다.

**Exception:** 동일 파일/동일 원인으로 묶이는 이슈는 한 커밋으로 처리할 수 있다.

---

### SQ-01-03. 루트 원인 문장을 먼저 확정한다

**Rule:** 코드 수정 전, 이슈별 루트 원인을 1~2문장으로 먼저 기록한다.

**Why:** 증상 기반 수정(guess fix)을 막고, 검증 기준(무엇이 해결인지)을 명확히 할 수 있다.

**DO:**

- `원인`, `영향`, `수정 가설`을 짧게 기록한다.
- 재현 가능한 이슈는 재현 절차를 함께 기록한다.

**Exception:** 규칙 단순 위반(예: 불필요 modifier, 명백한 API 오용)은 원인 문장을 1문장으로 축약할 수 있다.

---

### SQ-01-04. 검증 게이트를 강제한다

**Rule:** 모든 수정은 “테스트 + Sonar 재분석”을 통과해야 완료로 간주한다.

**Why:** 로컬 테스트 통과만으로는 Sonar 규칙 충족 여부를 보장할 수 없고, Sonar 통과만으로는 런타임 안정성을 보장할 수 없다.

**DO:**

- Frontend: 테스트 실행 후 Sonar 재분석
- Backend: 테스트(또는 verify) 실행 후 Sonar 재분석
- 재분석 후 대상 이슈 상태 변화(해결/잔존)를 확인한다.

**Exception:** 인프라 문제로 Sonar 재분석이 일시 불가하면, 테스트 결과와 미실행 사유를 기록하고 보류 상태로 닫는다.

---

### SQ-01-05. 처리 이력 문서 포맷을 고정한다

**Rule:** 각 배치는 동일한 템플릿(대상/원인/수정/검증/결과)으로 기록한다.

**Why:** 반복 작업에서 비교 가능성을 확보하고, 다음 사이클에서 누락/중복을 줄일 수 있다.

**DO:**

- `docs/issues/sonarqube/`에 날짜 기반 처리 문서를 생성한다.
- 각 항목에 이슈 키, 파일, 규칙 ID, 결과 상태를 기록한다.

**Exception:** 없음.

---

### SQ-01-06. 우선순위 외 이슈는 명시적으로 보류한다

**Rule:** 이번 사이클 범위 밖(MINOR/INFO 중심)은 “보류 목록”으로 별도 관리한다.

**Why:** 범위를 벗어난 작업이 섞이면 안정성 개선 속도가 떨어지고 완료 기준이 흔들린다.

**DO:**

- 보류 사유와 다음 검토 시점을 함께 기록한다.

**Exception:** 범위 밖 이슈라도 동일 변경에서 부수적으로 해결되면 함께 반영 가능하다.

---

### SQ-01-07. 지표 비교 기준을 고정한다

**Rule:** 배치 전/후 비교는 동일 메트릭(coverage, bugs, blocker, critical, major)을 사용한다.

**Why:** 매번 다른 지표를 쓰면 개선 효과를 정량적으로 판단할 수 없다.

**DO:**

- 전/후 지표 스냅샷을 같은 문서에 기록한다.

**Exception:** Sonar 버전 변경 등으로 지표 정의가 바뀐 경우 변경 사유를 명시한다.

---

## Conflict Resolution

- `SQ-01-01`(범위 고정)과 “빠른 정리” 요구가 충돌하면, 범위 고정을 우선한다.
- `SQ-01-02`(최소 변경)와 “파일 단위 일괄 수정”이 충돌하면, 안정성 영향도가 같은 항목만 묶어 처리한다.
- `SQ-01-04`(검증 게이트)와 시간 압박이 충돌하면, 미검증 상태로 완료 처리하지 않고 보류로 전환한다.

---

## Design Decisions

| 결정 | 근거 |
|------|------|
| 우선 범위를 `BUG + (BLOCKER/CRITICAL/MAJOR)`로 고정 | 안정성 중심 처리와 처리량 관리의 균형 |
| “테스트 + Sonar 재분석” 이중 게이트 채택 | 기능 안정성과 정적 규칙 충족을 동시에 보장 |
| 날짜 기반 문서 누적 방식 채택 | 회차별 추세 비교와 회귀 추적 용이 |
