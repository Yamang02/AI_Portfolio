# Logic-Level Rules

> Agent가 코드 생성 시 자동 참조하는 로직 레벨 규칙.
> 함수 설계, 에러 처리는 별도 문서에서 다룬다.
> 특정 언어/프레임워크에 종속되지 않는 보편 원칙이며, 스택별 문서가 이 규칙을 상속·확장한다.

---

## Meta

### 적용 범위

- 비즈니스 로직, 계산, 데이터 변환 등 **순수 로직 코드**에 적용한다.
- 인프라(DB, API, I/O), 프레임워크 보일러플레이트, 설정 코드에는 적용하지 않는다.

### 규칙 티어

| 티어 | 규칙 | 기준 | 최소 적용 |
|------|------|------|----------|
| **Core** | CC-01-01, 02, 03, 04, 08 | 위반 시 로직의 정확성·가독성이 직접 훼손됨 | S |
| **Guide** | CC-01-05, 06, 07, 09, 10 | 코드 품질에 영향을 주지만, 맥락에 따라 유연하게 적용 가능 | S |

### 우선순위 체계

규칙 간 충돌 시 아래 순서를 따른다:

1. **가독성** — 읽는 사람이 위에서 아래로 따라갈 수 있는가
2. **예측 가능성** — 같은 패턴이 같은 방식으로 반복되는가
3. **단순성** — 더 적은 개념으로 같은 결과를 낼 수 있는가

## Rules

### CC-01-01. 선형 흐름

**Rule:** 모든 로직은 위에서 아래로 한 방향으로 흐른다.

**Why:** 흐름이 점프하거나 역행하면 Agent와 사람 모두 실행 순서를 추적하기 어렵다. 검증 → 계산 → 변환 → 반환 순서가 역전되면 아직 검증되지 않은 값을 계산에 사용하는 등의 논리적 오류가 발생할 수 있다.

**DO:**
```pseudo
validate(input)
raw = calculate(input)
result = transform(raw)
return result
```

**Exception:** 없음. 이 규칙은 항상 적용한다.

---

### CC-01-02. Early Return

**Rule:** 예외 조건은 함수 상단에서 즉시 반환하여 본문 진입을 막는다.

**Why:** 본문이 항상 정상 흐름만 다루게 되어, 중첩 없이 평탄한 구조를 유지할 수 있다. 조건문 안에 본문을 감싸는 방식은 중첩을 유발하고 정상 흐름의 위치를 모호하게 만든다.

**DO:**
```pseudo
if input is empty -> return default
if input is invalid -> throw error

result = process(input)
return result
```

**Exception:** 조건이 하나뿐이고 본문이 1~2줄인 경우, 단순 if-else도 허용한다.

---

### CC-01-03. 중첩 깊이 제한

**Rule:** 조건문 중첩은 최대 1단계까지만 허용한다.

**Why:** 중첩이 깊어지면 각 분기의 전제 조건을 머릿속에 쌓아야 하므로 인지 부하가 급격히 증가한다. 2단계 이상 중첩이 필요하다면 조건을 분리하거나 함수를 추출해야 한다는 신호이다.

**DO:**
```pseudo
for item in items {
  if item.active -> total += item.price
}

for item in items {
  total += calculateItemPrice(item)
}
```

**Exception:** 없음. 중첩이 2단계 이상 필요하면 조건을 분리하거나 함수를 추출한다.

---

### CC-01-04. 상태의 단계적 변환

**Rule:** 하나의 값이 같은 의미를 유지하며 단계적으로 정제될 때, 동일 변수에 순서대로 재할당한다.

**적용 조건:** 변수명이 모든 재할당 시점에서 여전히 정확한 이름일 것. 재할당 후 이름이 값을 설명하지 못하면 CC-01-06에 따라 별도 변수를 사용한다.

**Why:** 변환 파이프라인이 한 변수 위에서 진행되면, 흐름을 위에서 아래로 따라가기만 하면 된다. 단계마다 새 변수를 만들면 이름 짓기 부담과 변수 추적 비용이 함께 증가한다.

**DO:**
```pseudo
total = sumItems(order.items)
total = applyDiscount(total, coupon)
total = applyTax(total, taxRate)
return total
```

**Exception:** 변환 전후 값을 모두 참조해야 하는 경우, 의미가 구분되는 별도 변수를 사용한다.

```pseudo
basePrice = getPrice(item)
discountedPrice = applyDiscount(basePrice, coupon)
savings = basePrice - discountedPrice   // 두 값 모두 필요
```

---

### CC-01-05. 변수명은 현재 상태를 표현

**Rule:** 변수명은 해당 시점에서 그 값이 무엇인지를 나타내야 한다.

**Why:** 이름이 상태를 정확히 반영하면, 코드를 읽을 때 별도 추적 없이 의미를 알 수 있다. `list`, `data` 같은 일반 이름은 어떤 맥락에든 들어맞기 때문에 오히려 아무 정보도 전달하지 못한다.

**DO:**
```pseudo
unpaidOrders = filterUnpaid(orders)
overdueOrders = filterOverdue(unpaidOrders)
```

**금지 이름:** `a`, `b`, `data`, `value`, `tmp`, `result2`, `list2`

**Exception:** 관용적으로 의미가 명확한 루프 인덱스(`i`, `j`)와 좌표(`x`, `y`)는 허용한다.

---

### CC-01-06. 변수 의미 고정

**Rule:** 한 변수에 서로 다른 의미의 값을 재할당하지 않는다.

**Why:** 같은 이름이 문맥에 따라 다른 의미를 가지면, 코드의 어느 지점에서든 "지금 이 변수가 뭐지?"를 다시 추적해야 한다.

**DO:**
```pseudo
rawInput = getUserInput()
sanitizedInput = sanitize(rawInput)
```

**CC-01-04와의 관계:** CC-01-04의 단계적 변환은 같은 의미(예: "주문 총액")가 정제되는 과정이다. CC-01-06은 의미 자체가 바뀌는 재할당을 금지한다. 판단 기준: **변수명이 여전히 정확한가?** 재할당 후에도 이름이 값을 정확히 설명하면 CC-01-04, 아니면 CC-01-06 위반이다.

**Exception:** 없음.

---

### CC-01-07. 컬렉션은 명령형 루프 우선

**Rule:** 컬렉션 순회는 명령형 루프를 기본으로 사용한다.

**Why:** 명령형 루프는 단계별 실행이 명시적이어서, Agent가 생성한 코드를 사람이 검증하기 쉽다. 체이닝이 길어지면 중간 상태를 확인하기 어렵고, 각 단계의 입출력을 머릿속에서 추적해야 한다.

**DO:**
```pseudo
totals = []
for item in items {
  totals.append(item.price * item.quantity)
}
```

**Exception:** 각 단계가 단일 연산(filter, map, sorted 등)이고 부수효과가 없는 순수 변환 체이닝은 허용한다. 중간 결과를 별도 변수로 확인해야 이해할 수 있다면 명령형으로 전환한다.

```pseudo
activeItems = items.filter(item -> item.active)              // 허용
names = items.filter(item -> item.active).map(item -> item.name)  // 허용
```

### CC-01-08. 순수 함수 — 외부 의존은 인자로, 동일 입력엔 동일 결과

**Rule:** 로직 함수는 모든 외부 의존(DB 조회 결과, API 응답, 설정값 등)을 인자로 받는다. 함수 내부에서 외부 시스템을 직접 호출하지 않으며, 같은 입력에 대해 항상 같은 결과를 반환한다.

**Why:** 외부 호출이 함수 안에 있으면 테스트·재사용 시 외부 의존성이 따라오고, 동일 입력에 다른 결과가 나올 수 있어 디버깅과 코드 추론이 어려워진다. 인자로 주입하면 로직 함수는 순수 계산만 담당하게 되어 입력-출력 관계가 명확하고 결과가 예측 가능해진다.

**DO:**
```pseudo
calculateMemberDiscount(total, memberGrade) {
  rate = getDiscountRate(memberGrade)
  return total * (1 - rate)
}

calculateTax(price, rate) {
  return price * rate
}
```

**Exception:** 없음. 외부 상태가 필요하면 인자로 주입한다.

---

### CC-01-09. 함수 복잡도 제한

**Rule:** 하나의 함수는 다음 네 가지 제한을 모두 만족해야 한다.

| 지표 | 제한 | 카운팅 기준 |
|------|------|------------|
| 책임 수 | 1개 | 함수가 수행하는 역할은 하나뿐이어야 한다 |
| 상태 변화 | 5회 이내 | 변수 선언·재할당 각 1회. 반복문 내 누적 연산은 루프 전체를 1회로 간주. 함수 호출 자체는 세지 않고 반환값을 받는 할당만 센다 |
| 분기 수 | 3개 이내 | if, switch의 각 분기를 1개로 센다. early return도 각각 1개의 분기이다 |
| 조건식 복잡도 | AND/OR 2개 이내 | 하나의 조건식에서 논리 연산자(&&, \|\|)는 최대 2개까지만 허용한다. 초과 시 조건을 변수로 추출하거나 함수로 분리한다 |

**Why:** LLM과 사람 모두 함수를 이해할 때 추적해야 하는 것은 줄 수가 아니라 상태 변화, 분기 수, 조건의 복잡도이다. 줄 수가 짧아도 분기가 많으면 복잡하고, 길어도 선형 파이프라인이면 단순하다. 또한 줄 수 기반 제한은 함수를 과도하게 쪼개게 만들어, 파일 수 증가와 탐색 비용 상승으로 이어질 수 있다.

**DO:**
```pseudo
processOrder(order, coupon) {
  if not order.items -> return empty
  total = sumItems(order.items)
  total = applyDiscount(total, coupon)
  total = applyTax(total, order.taxRate)
  return buildResult(order, total)
}

isEligible = isActive && hasCoupon
isValid = hasItems && isActive || isVIP
```

**Exception:** 없음. 제한을 초과하면 함수를 분리한다.

### CC-01-10. 변수 선언은 const 우선

**Rule:** 변수를 선언할 때 재할당이 필요하지 않다면 불변 바인딩(const, final, val 등)을 기본으로 사용한다.

**Why:** 불변 바인딩은 "이 변수는 이후 바뀌지 않는다"를 선언 시점에 확정한다. 코드를 읽을 때 값의 변경 가능성을 추적할 필요가 없어지므로 인지 부하가 줄어든다. Agent가 생성한 코드를 검증할 때도, 불변 바인딩은 해당 변수의 값이 할당 시점에 고정됨을 보장하므로 추론 범위가 좁아진다.

**DO:**
```pseudo
const tax = price * taxRate
const total = subtotal + tax
```

**CC-01-04와의 관계:** CC-01-04(단계적 변환)는 같은 의미가 정제되는 과정에서 재할당을 허용한다. 재할당이 필요한 변수는 가변 바인딩(let, var 등)을 사용하되, 그 외 모든 변수는 const를 기본으로 한다.

**Exception:** CC-01-04에 해당하는 단계적 변환, 루프 카운터 등 재할당이 본질적으로 필요한 경우.

## Conflict Resolution

규칙 간 충돌이 발생하면 아래 순서로 판단한다.

| 우선순위 | 판단 기준 | 설명 |
|---------|----------|------|
| 1 | 가독성 우선 | 두 규칙이 충돌하면, 위에서 아래로 읽기 쉬운 쪽을 택한다 |
| 2 | 규칙 번호 순 | 같은 가독성이면 낮은 번호(상위 규칙)가 우선한다 |
| 3 | Exception 확인 | 해당 규칙에 명시된 Exception에 해당하는지 먼저 확인한다 |

### 자주 발생하는 충돌

**CC-01-04 vs CC-01-06:** 재할당이 "같은 의미의 정제"인지 "다른 의미로의 변경"인지 판단.
→ 기준: 재할당 후에도 변수명이 값을 정확히 설명하면 CC-01-04 적용, 아니면 별도 변수를 사용.

**CC-01-09 vs CC-01-03:** 중첩을 제거하면 함수가 길어지는 경우.
→ 함수를 추출하여 두 규칙을 모두 만족시킨다.

**CC-01-07 vs 단순성:** 명령형 루프가 순수 변환 체이닝보다 오히려 복잡한 경우.
→ CC-01-07의 Exception에 따라 각 단계가 단일 연산이고 부수효과가 없는 순수 변환 체이닝을 허용한다.

**CC-01-03 + CC-01-09 분기 누적:** early return 3개와 중첩 조건이 함께 오면 분기가 CC-01-09 제한(3개)을 초과한다.
→ CC-01-09 게이트가 자동으로 걸러준다. 함수를 분리하여 두 규칙을 모두 만족시킨다.

**CC-01-10 vs CC-01-04:** const 우선 원칙과 단계적 변환의 재할당이 충돌하는 경우.
→ CC-01-04에 해당하는 단계적 변환은 가변 바인딩을 사용한다. 그 외 변수는 const를 기본으로 한다.

## Design Decisions

이 문서의 설계 결정과 근거를 기록한다. Agent 주입용 문서 작성 시 참고할 것.

### 문서 구조

| 결정 | 근거 |
|------|------|
| 이 문서는 한국어 사람용 원본이다 | Agent 주입용 문서는 별도로 영어로 작성 예정. 이 문서는 검토·관리용 원본 역할 |
| 규칙마다 ID(CC-01-01~CC-01-10) 부여 | Agent가 위반 사항을 참조할 때 "CC-01-03 위반" 식으로 문서+규칙을 한 번에 특정 가능 |
| Rule → Why → DO → Exception 포맷 통일 | Agent가 파싱하기 쉬운 일관된 구조. Why가 있어야 엣지케이스에서 원칙 기반 판단 가능 |
| 함수 설계, 에러 처리는 별도 문서로 분리 | 로직 레벨에만 집중하여 범위를 명확히 함 |

### DON'T 예시를 제거한 이유

LLM은 프롬프트에 포함된 코드 패턴을 "본 적 있는 것"으로 처리한다. DON'T 레이블을 달아도 해당 패턴이 컨텍스트에 존재하는 것 자체가 생성 확률을 높일 수 있다. 따라서 **DO 예시만 남기고, 금지 이유는 Why에서 자연어로 설명**하는 방식을 채택했다.

### Agent 주입용 문서 작성 시 고려사항

- **토큰 비용 최적화**: 코드 생성마다 주입되므로 길이가 곧 비용. `Exception: 없음` 반복 제거, Rule+Why만으로 충분한 규칙은 DO 예시 생략
- **언어 통일**: 규칙, 설명, 의사코드 모두 영어로 통일하여 언어 전환 비용 제거
- **티어링**: 10개 규칙을 핵심/보조로 나누면 Agent가 우선순위를 잡기 쉬움
- **Conflict Resolution 분리**: 충돌이 없는 상황에서도 Agent가 이 섹션을 읽고 불필요한 충돌 분석에 추론 토큰을 소비할 수 있으므로, 별도 참조 문서로 분리 검토
- **수치 기준은 이진 판단 가능하게**: CC-01-09를 복잡도 기반으로 변경하면서 줄 수 기반의 모호한 소프트 바운더리는 제거됨. 현재 수치 기준(상태 변화 5회, 분기 3개, AND/OR 2개)은 모두 이진 판단이 가능한 형태

### 체인 스타일을 채택하지 않은 이유

파이프라인 표현 시 체인 스타일(`.then(fn1).then(fn2)`)도 검토했으나, flat 재할당 스타일을 채택했다.

- **데이터 흐름이 암시적**: 체인은 각 단계 사이를 흐르는 값이 보이지 않아, LLM이 중간 상태를 추론해야 한다
- **복수 인자 처리 불가**: `applyDiscount(total, coupon)`처럼 외부 값이 필요한 단계에서 체인이 깨진다
- **스택 종속**: `.then`은 Promise/async 패턴으로 특정 언어에 종속된다

flat 스타일은 각 줄에서 입력과 출력이 명시적이므로, LLM이 토큰 단위로 코드를 생성할 때 다음 줄의 컨텍스트가 명확하다. 파이프라인 의도는 주석으로 전달할 수 있다.

### 네이밍 범위의 경계 (DDD 기준)

이 문서(CC-01)의 네이밍 규칙은 **순수 로직 함수 내부의 변수명**에만 적용된다. 그 외 모든 네이밍(메소드명, 타입명, 이벤트/커맨드명 등)은 네이밍 컨벤션 문서(CN-01)에서 통합 관리한다.

| 영역 | 네이밍 성격 | 예시 | 관리 위치 |
|------|-----------|------|----------|
| 순수 로직 변수 | 상태 표현, 의미 고정 | `unpaidOrders`, `total` | CC-01 (이 문서: CC-01-05, CC-01-06) |
| 메소드명, 타입명, 이벤트/커맨드 등 | 레이어별 책임·패턴 반영 | `calculateTotal`, `Order`, `OrderPlaced` | CN-01 (네이밍 컨벤션) |

CC-01에서 네이밍을 언급할 때(CC-01-05, CC-01-06 등)는 항상 순수 로직 함수 내부의 변수명 맥락임을 전제한다.

### CC-01-09를 줄 수 기반에서 복잡도 기반으로 변경한 이유

기존 규칙은 "20줄 이하, 최대 30줄"이라는 줄 수 기반 제한이었으나, 다음 문제로 인해 복잡도 기반으로 재작성했다.

- **줄 수는 복잡도와 비례하지 않는다**: 40줄 선형 파이프라인은 단순하고, 10줄짜리 분기 폭탄은 복잡하다. LLM이 실제로 어려워하는 것은 줄 수가 아니라 추적해야 할 상태 변화와 분기 경로의 수이다
- **과도한 파일 분할을 유발한다**: 줄 수 제한을 엄격히 적용하면 함수가 잘게 쪼개지고, 파일 수가 증가하여 Agent의 인덱싱 비용과 컨텍스트 소비가 커진다
- **모호한 소프트 바운더리**: "약 20줄", "최대 30줄" 같은 표현은 Agent가 매번 다르게 해석한다. 반면 "상태 변화 5회 이내", "분기 경로 3개 이내"는 이진 판단이 가능하다
- **모델·스택 비종속**: 줄 수는 언어의 문법 밀도에 따라 의미가 달라지지만, 상태 변화와 분기 경로는 언어에 무관하게 일정한 복잡도 지표이다

### Lint 자동화 가능 범위

규칙을 검증하는 방식은 Lint(기계적 판단)와 Agent 리뷰(의미적 판단)로 나뉜다. Lint 규칙 설계 시 참고할 것.

| 규칙 | Lint 자동화 | 근거 |
|------|-----------|------|
| CC-01-03. 중첩 깊이 제한 | 가능 | AST 기반 depth 카운팅 (`max-depth` 등 기존 규칙 활용) |
| CC-01-09. 상태 변화 | 가능 | 변수 선언·재할당 카운팅은 AST로 정확히 셀 수 있음 |
| CC-01-09. 분기 수 | 가능 | if/switch/early return 카운팅. cyclomatic complexity와 유사 |
| CC-01-09. 조건식 복잡도 | 가능 | AND/OR 연산자 수 카운팅 |
| CC-01-01. 선형 흐름 | 부분적 | goto/break 정도만 감지 가능. 순서 역전은 의미적 판단 필요 |
| CC-01-04. 상태의 단계적 변환 | 불가능 | "같은 의미의 정제인지"는 기계적으로 판단 불가 |
| CC-01-05. 변수명 표현력 | 불가능 | 금지 이름 목록은 Lint로 가능하나, 이름의 적절성은 의미적 판단 |
| CC-01-06. 변수 의미 고정 | 불가능 | "의미가 바뀌었는지"는 사람/LLM 판단 영역 |
| CC-01-08. 순수 함수 | 부분적 | I/O 함수 호출 감지는 가능하나, 모든 외부 의존 식별 및 순수성 보장은 정적 분석으로 완전히 검증 불가 |
| CC-01-10. const 우선 | 가능 | `prefer-const` 등 기존 Lint 규칙으로 재할당 없는 let/var 감지 가능 |

수치 기반 규칙은 Lint로, 의미 기반 규칙은 Agent 리뷰로 커버하는 2단 구조를 권장한다.
